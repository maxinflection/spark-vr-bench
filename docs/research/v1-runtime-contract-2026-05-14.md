## 2. Agent-Task Invocation Surface

**V1 completely replaces the V0 CLI model.** There is no `python run.py --task_id ...` invocation in V1. The architecture is HTTP/API-driven.

### Two-tier HTTP API

**Tier 1: Caller → App-server** (port 3000)

The caller interacts with the app-server REST API. Routes are under `/api/v1` prefix.
- Source: `https://raw.githubusercontent.com/All-Hands-AI/OpenHands/main/openhands/app_server/v1_router.py`
- Source: `https://raw.githubusercontent.com/All-Hands-AI/OpenHands/main/openhands/app_server/app_conversation/app_conversation_router.py`

Key endpoints:

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/api/v1/app-conversations` | Start conversation; returns `AppConversationStartTask` with `status`, `app_conversation_id`, `sandbox_id`, `agent_server_url` |
| `POST` | `/api/v1/app-conversations/stream-start` | Streaming startup; returns streaming JSON of `AppConversationStartTask` objects |
| `GET` | `/api/v1/app-conversations/{id}` | Get conversation info including `execution_status` |
| `POST` | `/api/v1/app-conversations/{id}/send-message` | Send follow-up message; requires sandbox in `RUNNING` state |
| `DELETE` | `/api/v1/app-conversations/{id}` | Delete conversation |
| `POST` | `/api/v1/settings` | Set LLM/agent settings (deep-merge); where `api_key` and `model` are configured |
| `POST` | `/api/v1/settings/profiles/{name}` | Save named LLM profile |

**`AppConversationStartRequest` fields** (source: `app_conversation_models.py`):
- `llm_model: str | None` — model identifier string (e.g. `"anthropic/claude-sonnet-4-5-20250929"`)
- `initial_message` — `SendMessageRequest` with role + content blocks
- `agent_type` — `AgentType.DEFAULT` or `AgentType.PLAN`
- `secrets: dict[str, SecretStr] | None` — API keys and secrets passed per-conversation
- `sandbox_id`, `conversation_id`, `selected_repository`, `plugins`, `processors`, etc.

**Tier 2: App-server → Agent-server** (port 8000)

The app-server makes HTTP calls to agent-server containers on the caller's behalf.
- Source: `https://raw.githubusercontent.com/All-Hands-AI/OpenHands/main/openhands/app_server/app_conversation/live_status_app_conversation_service.py`

| Method | Agent-server path | Purpose |
|---|---|---|
| `POST` | `/api/conversations` | Start LLM agent conversation; body: `StartConversationRequest` with `agent` (LLM + tools), `workspace`, `initial_message` |
| `POST` | `/api/acp/conversations` | ACP agent variant |
| `GET` | `/api/conversations?ids=...` | Batch-fetch conversation status (execution_status polling) |
| `PATCH` | `/api/conversations/{id}` | Update conversation metadata |
| `DELETE` | `/api/conversations/{id}` | Delete conversation |
| `POST` | `/api/conversations/{id}/events` | Send messages/events |
| `GET` | `/api/conversations/{id}/agent_final_response` | Get agent's final response text (FinishAction or last MessageEvent) |
| `POST` | `/api/conversations/{id}/run` | Execute conversation in background |
| `POST` | `/api/conversations/{id}/switch_llm` | Swap LLM instance mid-conversation |

**Auth**: `X-Session-API-Key` header is required on all agent-server routes except `/health` and `/ready`.
- Source: `https://raw.githubusercontent.com/All-Hands-AI/OpenHands/main/openhands/app_server/app_conversation/live_status_app_conversation_service.py`

**`StartConversationRequest`** (agent-server level, source: `openhands-sdk`):
```
{
  workspace: LocalWorkspace(working_dir=...),
```

## 3. Privilege and Network Model

### Privilege

**No `--privileged` flag**. The docker_sandbox_service creates agent-server containers without privileged mode.
- KVM support: if `kvm_enabled=true`, `/dev/kvm:/dev/kvm:rwm` device is added.
- Source: `docker_sandbox_service.py` (cited above)

DinD is not required for the agent-server container itself. The app-server container requires Docker socket access (`/var/run/docker.sock`) to spin up sandboxes.

### Network model

V1 **inverts** the V0 network model significantly:

| Aspect | V0 | V1 |
|---|---|---|
| Docker socket | Not required by run.py | Required by app-server container |
| Network mode | `use_host_network=False` (bridge); agent reaches host via docker0 gateway | App-server can use bridge or host (`OH_SANDBOX_USE_HOST_NETWORK` env, added v1.3.0) |
| host.docker.internal | Not used | `SANDBOX_LOCAL_RUNTIME_URL=http://host.docker.internal` baked into app-server image |
| Agent→host path | `CYBERGYM_SERVER_URL_FOR_AGENT` = docker0 gateway IP | Agent-server's `WEBHOOK_CALLBACK_VARIABLE` = `http://host.docker.internal:{port}/api/v1/webhooks` |
| Task invocation | CLI (caller invokes subprocess) | HTTP POST to app-server (caller makes REST call) |
| Container lifetime | auto_remove=true per task (V0 runtime) | Sandbox lifecycle managed by app-server; dedicated volume `openhands-workspace-{id}` created/removed per conversation |

**Cybergym-specific concern**: The cybergym.server grading sidecar still needs to be reachable from within the agent-server container. The docker0 gateway URL problem from V0 (`CYBERGYM_SERVER_URL_FOR_AGENT`) likely persists in V1 — the agent-server container is on a bridge network and needs to reach the host's cybergym.server at its gateway IP.

Under V1, the cybergym server URL would need to be passed to the agent as part of the `initial_message` task description or via a workspace file, since there is no CLI `--server` flag equivalent in the HTTP API. [UNVERIFIED — no upstream cybergym V1 example confirms this.]

### Host connectivity

The app-server adds the host gateway via `extra_hosts: ["host.docker.internal:host-gateway"]` in docker-compose.yml. The agent-server container has `WEBHOOK_CALLBACK_VARIABLE` pointing to `http://host.docker.internal:{host_port}/api/v1/webhooks`.
- Source: `docker-compose.yml`, `docker_sandbox_service.py` (both cited above)

## 4. Token Usage Telemetry

### Where token usage lives in V1

V1 does track token usage, but **not in a flat result.json file**. The V0 pattern (grep `run.log` or scrape `verdict.json`) does not apply.

**Data flow**:
1. Agent-server `LLM` class calls litellm; response is processed by `Metrics` class
2. Per-call `TokenUsage` is accumulated into `Metrics.accumulated_token_usage`
3. `ConversationInfo.stats` (`ConversationStats`) aggregates metrics by `usage_id` across all LLM calls
4. `ConversationInfo.metrics` holds a `MetricsSnapshot`

**`TokenUsage` fields** (source: `openhands-sdk/openhands/sdk/llm/utils/metrics.py`):
- `prompt_tokens: int` (default 0)
- `completion_tokens: int` (default 0)
- `cache_read_tokens: int` (default 0)
- `cache_write_tokens: int` (default 0)
- `reasoning_tokens: int` (default 0)
- `context_window: int` (default 0)
- `per_turn_token: int` (default 0)
- `response_id: str`
- `model: str`

**`MetricsSnapshot` fields**:
- `model_name: str` (default "default")
- `accumulated_cost: float` (default 0.0)
- `max_budget_per_task: float | None`
- `accumulated_token_usage: TokenUsage | None`

**Retrieval path**: After conversation completes (`execution_status == FINISHED/ERROR/STUCK`):
- Call `GET /api/conversations/{id}` on the agent-server → `ConversationInfo.metrics.accumulated_token_usage.{prompt_tokens, completion_tokens}`
- Or access via `ConversationStats.get_combined_metrics()` (SDK pattern, see example `01_convo_with_local_agent_server.py`)

**SDK example** (primary source):
```python
cost = conversation.conversation_stats.get_combined_metrics().accumulated_cost
```
- Source: `https://raw.githubusercontent.com/OpenHands/software-agent-sdk/main/examples/02_remote_agent_server/01_convo_with_local_agent_server.py`

**Dashboard math impact**: Our current `result.json` has `tokens_in: 0, tokens_out: 0` hardcoded (the current run.py path doesn't extract them). V1 does expose these via the HTTP API, but requires explicit extraction from `ConversationInfo.metrics` after task completion. The data IS available — it just requires a new extraction step.

## 5. Bedrock Cross-Region Inference Profile Support

### LLM class model string handling

The SDK `LLM` class passes model strings directly to litellm. It infers the provider via `litellm.get_llm_provider()` (delegated entirely to litellm).
- Source: `https://raw.githubusercontent.com/OpenHands/software-agent-sdk/main/openhands-sdk/openhands/sdk/llm/llm.py`

For Bedrock, the LLM class has explicit handling:
```python
if api_key_value is not None and self._infer_litellm_provider() == "bedrock":
    return None  # Don't forward non-Bedrock API keys to Bedrock (uses IAM/SigV4)
```

This means `bedrock/us.anthropic.claude-opus-4-7-v1:0` format should route correctly if litellm recognizes it.

### Temperature stripping for Opus 4.7

V1 SDK uses `get_features(llm.model).supports_extended_thinking` to decide whether to strip temperature.
- `EXTENDED_THINKING_MODELS` list (source: `model_features.py`): `["claude-sonnet-4-5", "claude-sonnet-4-6", "claude-haiku-4-5"]`
- **`claude-opus-4-7` is NOT in this list.**
- Source: `https://raw.githubusercontent.com/OpenHands/software-agent-sdk/main/openhands-sdk/openhands/sdk/llm/utils/model_features.py`

**Critical finding**: The `get_features()` model matching uses case-insensitive substring search on the raw model string. For `bedrock/us.anthropic.claude-opus-4-7-v1:0`, the substring `claude-opus-4-7` IS present. However, since `claude-opus-4-7` is not in `EXTENDED_THINKING_MODELS`, the extended-thinking temperature strip does NOT fire.

Temperature stripping for reasoning models (source: `chat_options.py`):
- Temperature is stripped `if get_features(llm.model).supports_reasoning_effort and llm.reasoning_effort is not None`
- It is also stripped for extended thinking models
- Neither path applies to `claude-opus-4-7` as defined in the current `model_features.py`

**Result**: V1 SDK will pass `temperature` through to litellm for `bedrock/us.anthropic.claude-opus-4-7-v1:0`. Whether litellm 1.83.x+ handles the temperature strip for Bedrock Opus 4.7 then determines whether the call succeeds. The matrix doc noted this as [UNVERIFIED] and it remains so here.

**Bedrock cross-region profile format**: The `bedrock/` prefix is recognized by litellm provider inference. Cross-region profiles with `us.anthropic.*` in the model ID are a litellm concern, not the SDK's. [UNVERIFIED that litellm 1.83.14 correctly handles `bedrock/us.anthropic.claude-opus-4-7-v1:0` — requires a smoke test.]

## 6. max_iter and Context-Window Controls

### max_iterations

V1 exposes `max_iterations` as a first-class field on `StartConversationRequest`.
- Source: `https://raw.githubusercontent.com/OpenHands/software-agent-sdk/main/openhands-sdk/openhands/sdk/conversation/request.py`
- Default: 500 (minimum: 1)
- Set per-conversation at creation time in the `POST /api/conversations` body
- `ConversationInfo` also has a `max_iterations` field confirming it persists on the conversation object

**V0 analogue**: `--max_iter` CLI flag to `run.py`. In V1 this is `max_iterations` in the `StartConversationRequest` JSON body.

### Context-window / condenser

V1 has a full condenser framework. Types: `"noop"`, `"observation_masking"`, `"recent"`, `"llm"`, `"amortized"`.
- Source: `https://raw.githubusercontent.com/All-Hands-AI/OpenHands/main/config.template.toml`

In V0 we patched `max_model_len` via `keep_first=4` in the condenser config. In V1:
- Condenser is configured via the app-server settings (`[condenser]` section in `config.template.toml`)
- The condenser strategy and its parameters are set in server config or per-agent settings
- [UNVERIFIED: whether the V1 condenser has an equivalent `keep_first` parameter]
- `max_input_tokens` and `max_output_tokens` are fields on the `LLM` class in the SDK
  - Source: `https://raw.githubusercontent.com/OpenHands/software-agent-sdk/main/openhands-sdk/openhands/sdk/llm/llm.py`

**Context-window feedback issue (bd memory)**: The V0 bug was that `max_model_len=8192` caused overflow. In V1, `max_input_tokens` on the `LLM` object and condenser settings serve analogous purposes. The correct place to set context limits in V1 is the `LLM(max_input_tokens=..., max_output_tokens=...)` constructor, not a runtime flag.

### LLM params summary (V1 SDK LLM class fields)

| V0 mechanism | V1 equivalent |
|---|---|
| `--max_iter N` CLI | `max_iterations=N` in `StartConversationRequest` |
| condenser `keep_first=4` | condenser config in app-server settings; type `"recent"` or `"amortized"` |
| `max_model_len` vLLM server | `LLM(max_input_tokens=...)` passed at conversation start |
| `--model bedrock/...` CLI | `LLM(model="bedrock/...")` in `Agent` in `StartConversationRequest` |

## 7. Auto-Remove and Cleanup Semantics

### V0 behavior

V0 used `auto_remove=true` in `config.toml` — the runtime container cleaned itself up after each task. The runner script hardcodes `CYBERGYM_OPENHANDS_RUNTIME_IMAGE` and pre-pulls once per session.

### V1 behavior

V1 uses a sandbox lifecycle managed by the app-server, not per-task auto-remove.

**Container lifecycle**:
- App-server creates a sandbox container (agent-server image) per conversation via `docker_sandbox_service.py`
- On `DELETE /api/v1/app-conversations/{id}`, the app-server tears down the sandbox container
- Named volume `openhands-workspace-{sandbox_id}` is also removed on deletion
- Source: `docker_sandbox_service.py` (cited above)

**Session persistence**: V1 supports **sandbox grouping strategies**: `NO_GROUPING`, `GROUP_BY_NEWEST`, `LEAST_RECENTLY_USED`, `FEWEST_CONVERSATIONS`, `ADD_TO_ANY`.
- Source: `settings_models.py`: `sandbox_grouping_strategy: SandboxGroupingStrategy`

This means V1 can reuse a running agent-server container across multiple conversations — the "auto-remove per task" behavior of V0 is NOT the default in V1. The caller must explicitly delete conversations to reclaim resources.

**Implication for our runner**: The V0 pattern of "one container per task, auto-removed" maps to V1's `DELETE /api/v1/app-conversations/{id}` after each task completes. The runner must track conversation IDs and explicitly delete them to avoid container accumulation.

## Headline Summary

V1 is a **fundamentally different architecture** from V0. The core change: task invocation moved from CLI subprocess (`python run.py --task_id ...`) to HTTP REST API (`POST /api/v1/app-conversations`). The app-server orchestrates everything — it spins up agent-server Docker containers per conversation, forwards `LLM_*` env vars into them, and exposes a polling API for conversation status. No `--privileged` flag needed. Token usage is available via `ConversationInfo.metrics.accumulated_token_usage` after the conversation reaches a terminal state (`FINISHED/ERROR/STUCK`). `max_iterations` is a first-class request field (default 500). Auto-remove is gone — caller must DELETE conversations explicitly. The V0 `examples/agents/openhands/run.py` entrypoint that our `run-pool-a-cybergym.sh` calls does not exist in V1; the entire Pool A runner must be rewritten as an HTTP API client.

## Surprises vs V0

1. **CLI entrypoint eliminated.** `run.py --task_id` is gone. All task invocation is via `POST /api/v1/app-conversations`. This is the single largest wiring rewrite.

2. **Two-container architecture.** The caller must run the app-server container (port 3000) AND it spins up agent-server containers (port 8000) dynamically. V0 was one container per task.

3. **LLM settings live in the app-server settings store, not per-task CLI.** To set API key + model for a run, the caller must either: (a) `POST /api/v1/settings` before conversations, or (b) pass `secrets` dict and `llm_model` string in `AppConversationStartRequest`. The `LLM_API_KEY` environment variable is NOT read by the settings router — it is forwarded as `LLM_*` prefix var into agent-server containers only if set on the app-server.

4. **Session API key per sandbox.** Each agent-server container requires `X-Session-API-Key` auth on all API calls. This key is auto-generated by the app-server (base62, 32 bytes) and injected into the container. Callers talking directly to the agent-server (as opposed to through the app-server) need to obtain this key.

5. **Auto-remove gone.** Each task conversation must be explicitly deleted. Without DELETE calls, agent-server containers accumulate.

6. **No cybergym V1 example exists upstream.** We would be the first to wire cybergym against the V1 HTTP API. The `--server` flag (cybergym grading URL) needs a new threading path — likely embedded in the task prompt or via workspace files, since there's no CLI flag equivalent.

7. **Temperature stripping for Opus 4.7 is NOT automatic in V1 SDK.** `claude-opus-4-7` is absent from `EXTENDED_THINKING_MODELS`. Temperature passes through to litellm. Whether litellm 1.83.14+ handles this for Bedrock is [UNVERIFIED].

## Risks for the Plan

1. **Silent failure analog to pgf Stage 1.** The app-server + agent-server + SDK are three separately versioned packages. Running mismatched versions will likely cause container exit after "Runtime-ready" handshake — exactly the V0 failure mode. Mitigation: pin all three to the same release (currently `1.21.1`).

2. **Bedrock cross-region profile temperature.** `bedrock/us.anthropic.claude-opus-4-7-v1:0` will have temperature passed through by the SDK. If litellm 1.83.14 does not strip it for Bedrock Opus 4.7, every call will fail with `InvalidParameterValue`. Dashboard will show 0 LLM calls. Mitigation: smoke test before committing V1 implementation.

3. **Token usage not in result.json by default.** Our current result.json hardcodes `tokens_in: 0, tokens_out: 0`. In V1, token counts require an explicit API call to `GET /api/conversations/{id}` after task completion and extraction of `metrics.accumulated_token_usage`. If this step is omitted, dashboard math breaks silently.

4. **cybergym server URL threading.** The `--server` CLI flag (cybergym grading endpoint) has no V1 equivalent. The URL must be embedded in the task prompt or passed via the workspace. If misconfigured, every PoC submission fails silently (0 poc_records rows) — same failure mode as the docker0 bug caught 2026-05-14.

5. **Container accumulation.** Without explicit DELETE calls per task, agent-server containers pile up. On a constrained harness instance, this exhausts memory and ports. The runner must track and delete conversation IDs.

6. **App-server requires Docker socket.** Running the app-server in a restricted environment (e.g., EC2 without Docker-in-Docker) requires the Docker socket to be available. If the harness EC2's Docker daemon is unavailable or the socket path differs, the app-server cannot create sandboxes.

7. **Sandbox grouping strategy.** Default grouping behavior is [UNVERIFIED]. If the app-server reuses agent-server containers across tasks (non-`NO_GROUPING` strategy), task isolation breaks — agent state from task N bleeds into task N+1.

8. **Context window / condenser V1 equivalent.** We tuned `keep_first=4` in V0. The V1 condenser config is in the app-server's settings store. Whether a pre-flight condenser config is necessary and what the correct V1 incantation is remains [UNVERIFIED].

## Recommended Pre-Flight Smokes

1. **Container start smoke (no LLM).** Pull `ghcr.io/openhands/agent-server:1.21.1-python`. Start with a known `SESSION_API_KEY`. Hit `GET /health` → expect `{"status":"ok"}`. Hit `GET /ready` → expect `{"status":"ready"}` within 30s. Confirm no exit after handshake.

2. **App-server → agent-server spawn smoke.** Start app-server container with Docker socket mounted. Verify it successfully spins up an agent-server container. Check `docker ps` shows the agent-server image. Confirm the sandbox shows `RUNNING` status via app-server API.

3. **Bedrock temperature smoke.** Submit a minimal `POST /api/conversations` with `LLM(model="bedrock/us.anthropic.claude-opus-4-7-v1:0", ...)` and `initial_message="What is 2+2?"`. Confirm the call reaches Bedrock (CloudTrail or server log), completes without `InvalidParameterValue`, and returns a non-empty response.

4. **Token usage extraction smoke.** After the Bedrock smoke above, `GET /api/conversations/{id}` and verify `metrics.accumulated_token_usage.prompt_tokens > 0` and `completion_tokens > 0`.

5. **cybergym server URL smoke.** Start the cybergym.server on the harness host (port 8666). From within a running agent-server container (exec into it), attempt `curl http://{docker0_ip}:8666/health` (or whatever path the cybergym server exposes). Confirm the agent-server container can reach the grading server via the docker0 gateway — this is the V1 analog of the V0 silent bug caught 2026-05-14.

6. **max_iterations smoke.** Submit a task with `max_iterations=3`. Verify the conversation terminates at or before 3 iterations and reports `STUCK` or `FINISHED`, not running indefinitely.

7. **DELETE/cleanup smoke.** Submit two tasks sequentially. After each, DELETE the conversation. Verify via `docker ps` that agent-server containers are removed. Confirm no container accumulation.
